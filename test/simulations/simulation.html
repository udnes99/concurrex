Executor – Adaptive Concurrency Simulation

CoDel queue management + throughput-driven AIMD concurrency regulation

Steady State

Baseline: 10, delayThreshold: 100ms, controlWindow: 100ms. 0–2000ms: 400 tasks arrive (mean inter-arrival 5ms) each taking ~20ms. Load stays within baseline capacity — in-flight hovers around 4–8, queue stays at 0. No CoDel activation, no throughput degradation. This is what normal operation looks like.

Shaded red = CoDel dropping (load shedding) · Shaded orange = throughput degraded (MD active)

Burst Absorption

Baseline: 10, delayThreshold: 100ms, controlWindow: 100ms. 0–1000ms: normal load, 100 tasks arriving every ~10ms, each taking ~20ms. ~1000ms: sudden burst — 200 tasks dumped at once. Queue spikes to ~190. AI grows the concurrency limit (+1 per completion while queue has work) to absorb the burst. 1000–6000ms: burst drains, no new arrivals. Baseline gravity snaps the limit back down as in-flight tasks complete. Key point: a long queue with fast processing is healthy — no CoDel drops, no MD.

Shaded red = CoDel dropping (load shedding) · Shaded orange = throughput degraded (MD active)

Throughput Degradation & Recovery

Baseline: 10, min: 2, delayThreshold: 500ms, controlWindow: 100ms. 0–1500ms: healthy phase — 150 tasks arriving every ~10ms, each taking ~15ms. EWMA warms up. ~1500–3500ms: backend degrades — tasks slow to ~200ms (arrival slows to ~40ms). Throughput monitor detects the completion rate drop and MD progressively cuts the concurrency limit. EWMA is frozen during the first ESS degraded windows, giving MD time for corrective action. ~3500–6000ms: backend recovers — tasks return to ~15ms. Queue drains, baseline gravity restores the limit. CoDel may activate briefly if MD cuts capacity below arrival rate.

Demand Spike (Healthy Processing)

Baseline: 10, min: 2, delayThreshold: 50ms, controlWindow: 100ms. 0–500ms: baseline established with 50 tasks arriving every ~10ms, each taking ~10ms. ~500ms: 500 tasks dumped at once. Queue spikes to ~490. Processing speed unchanged (~10ms). 500–10000ms: AI grows the limit as tasks complete with queue work. CoDel drops stale entries whose sojourn exceeds 50ms. Key point: MD does NOT fire — throughput (completions/window) stays healthy. A long queue is a demand problem, not a processing problem. The cashier doesn't slow down because the line is long.

Degraded Burst (CoDel Only)

Baseline: 10, min: 2, delayThreshold: 50ms, controlWindow: 100ms. 0–500ms: baseline established with 50 tasks, same as the healthy demand spike. ~500ms: 200 slow tasks (~150ms each) dumped at once. Queue spikes to ~190. 500–5500ms: AI grows the limit (tasks complete with queue work), CoDel drops stale entries. MD does NOT fire — the short baseline phase (~5 windows) doesn't accumulate enough variance samples for detection (requires ESS ≈ 9). This is a one-shot burst, not sustained degradation. Compare with the healthy demand spike: same shape, but slower processing means CoDel drops more aggressively. 5500–11000ms: 100 fast recovery tasks (~10ms) drain remaining queue.

Full Overload (CoDel + MD)

Baseline: 10, min: 2, max: 15, delayThreshold: 100ms, controlWindow: 100ms. 0–1500ms: healthy phase — 150 tasks arriving every ~10ms, each taking ~15ms. EWMA warms up (~15 windows). ~1500–3500ms: sustained backend degradation — tasks slow to ~300ms, arrival unchanged at ~10ms. Capacity drops well below arrival (100/sec). AI grows the limit but hits the cap (15) — not enough to compensate. Queue grows, entries age past 100ms. Both mechanisms activate: MD detects the throughput drop and cuts concurrency to reduce backend pressure. CoDel detects stale queue entries and sheds excess demand. ~3500–5500ms: backend recovers — tasks return to ~15ms. CoDel exits dropping, baseline gravity and AI restore the concurrency limit.
